# Trrip — AI-Powered Travel Itinerary Generator

A MERN-stack web application that turns travel booking documents (flight tickets, hotel confirmations, travel tickets) into structured, day-by-day AI-generated itineraries.

Built as a submission for the Junior Full Stack Developer (MERN + AI) screening assignment.

## Features

- **JWT Authentication** — secure register/login with bcrypt password hashing
- **Document Upload** — drag-and-drop PDFs or images (flight tickets, hotel bookings)
- **Text Extraction** — `pdf-parse` for PDFs, Tesseract.js OCR for images
- **AI Itinerary Generation** — Google Gemini (2.5 Flash) converts extracted text into a structured day-by-day plan
- **Itinerary History** — paginated list of all itineraries generated by the logged-in user
- **Sharing** — toggle any itinerary public/private; public itineraries get a unique shareable link viewable without login
- **Responsive UI** — built with React + Tailwind, works across mobile, tablet, and desktop

## Bonus Features Implemented

- Drag-and-drop upload (`react-dropzone`)
- Polished, custom UI/UX (not default Tailwind/Bootstrap look) with animations and a distinct visual identity
- Delete itinerary (cleans up both DB record and stored file)
- Upload progress indicator + generation status stages
- Public share page with its own branded layout (separate from the authenticated app)

### Note on AWS S3
The codebase includes a complete AWS S3 integration path (`config/multer.js` S3 storage engine, `services/s3Service.js` for deletion) but it is **not wired into the active code path** for this submission — file storage currently uses local disk (`/uploads`). This was a deliberate call: AWS account verification required billing/card details, which felt unnecessary risk for a take-home assignment. Swapping back to S3 is a one-file change (point `routes/uploadRoutes.js` at the S3-backed multer config) plus adding AWS credentials to `.env`.

## API Endpoints

| Method | Route | Auth | Description |
|--------|-------|------|-------------|
| POST | `/api/auth/register` | No | Create account |
| POST | `/api/auth/login` | No | Login, returns JWT |
| GET | `/api/auth/me` | Yes | Current user profile |
| POST | `/api/upload` | Yes | Upload document → extract → generate itinerary |
| GET | `/api/itineraries` | Yes | Paginated list of user's itineraries |
| GET | `/api/itineraries/:id` | Yes | Single itinerary (owner only) |
| PATCH | `/api/itineraries/:id/share` | Yes | Toggle public/private |
| GET | `/api/itineraries/share/:token` | No | View a public itinerary |
| DELETE | `/api/itineraries/:id` | Yes | Delete itinerary + associated file |

## Database Schema (high level)

- **User** — name, email (unique), hashed password
- **Upload** — original file metadata, extracted text, status (uploaded/extracting/extracted/failed), linked to User
- **Itinerary** — title, destination, summary, structured `days[]` (each with activities array), flight/hotel detail objects, `shareToken` (UUID), `isPublic` flag, status (generating/completed/failed), linked to User and Upload

## Problem-Solving Notes

- **Unpredictable AI output length**: Gemini occasionally returned itineraries long enough to get truncated mid-JSON. Fixed by raising `maxOutputTokens`, setting `responseMimeType: "application/json"`, and adding a regex-based fallback extractor in case the model still wraps output in prose or markdown fences.
- **Partial booking data**: many real-world tickets only contain a destination name with no explicit dates or hotel info. The AI prompt explicitly instructs the model to fill gaps with sensible, destination-appropriate suggestions rather than failing — this matches how a real travel-planning product would behave for an incomplete booking.
- **Image-based tickets**: since uploads can be screenshots/photos rather than clean PDFs, OCR preprocessing (greyscale + normalize via `sharp`) was added before running Tesseract to improve extraction accuracy.
